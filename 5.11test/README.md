# 🚀 Aero-Twin: 基于 RK3588 的模块化自动驾驶视觉系统

> **项目概述**：本项目是为 Orange Pi 5 (RK3588) 深度优化的自动驾驶与视觉感知系统。全面压榨 NPU 三核算力，集成了基于共享内存的低延迟推流、YOLO 目标检测、多叉树 (Beam Search) 路径规划、OCR 意图识别以及串级 PID 下位机控制。

## 📁 1. 目录结构与核心功能说明

```text
.
├── config.py              # ⚙️ 全局配置中心 (所有可调参数都在这里)
├── main.py                # 🧠 调度枢纽 (管理线程、共享内存、串口和 Web 推流)
├── models/                # 📦 模型仓库
│   ├── det/yolov8_1.rknn                 # 目标检测模型 (金币/路标)
│   ├── ocr/ppocrv4_rec_fp16.rknn         # OCR 识别模型 (仅 Rec)
│   └── seg/ppliteseg_576_final_int8.rknn # 赛道语义分割模型
├── modules/               # 🧩 业务逻辑模块 (核心算法)
│   ├── detector.py        # YOLO 推理与 NMS 过滤
│   ├── segmentor.py       # PPLiteSeg 推理、逆透视(BEV)、多叉树路径搜索、曲线拟合
│   └── ocr_system.py      # 接收 YOLO 的 ROI 进行文本识别
└── utils/                 # 🛠️ 工具箱
    ├── image_proc.py      # 图像处理 (如透视变换、缩放)
    └── visualize.py       # 绘图工具 (画框、画轨迹线)
   📄 2. 各模块文件详细解析
main.py (主编排引擎)

    功能：程序的入口。负责拉起所有的后台线程。

    包含的线程：

        ai_producer_thread: 从共享内存 (shm_ar_video) 零拷贝拉取最新视频帧，并分发给 Seg 和 YOLO 队列。

        yolo_and_ocr_worker: 消费者线程，跑 YOLO 检测，拿到框后立刻切图送给 OCR 识别，并根据文字（如 LEFT/RIGHT）修改全局转向意图。

        seg_worker: 消费者线程，跑分割和路径搜索，计算出偏差 err_x 和斜率 l_k，并渲染带有诊断信息的最终画面。

        serial_control_thread: 100Hz 高频线程，死循环读取 PID 误差并打包发送给下位机。

        Flask Web Server: 提供 HTTP 视频流预览 (http://<IP>:5003)。

modules/segmentor.py (最复杂的心脏)

    功能：处理“路在哪里”和“车怎么走”。

    核心流程：

        推理与掩膜：生成 0/1 赛道掩膜。

        逆透视 (BEV)：利用 cv2.warpPerspective 将前视画面拉平成鸟瞰图。

        多叉树搜索 (Beam Search)：在鸟瞰图中，向 9 个方向像树枝一样探索，筛选出最平滑的 30 条候选路径。

        断层/分叉检测：如果发现赛道中间有巨大空白（横向坐标差大于阈值），则判断为分叉路口，并根据 turn_intent (转向意图) 剪枝。

        控制量输出：对选中的路径点进行多项式拟合，计算出中心偏移量 err_x 和前瞻斜率 l_k。

⚙️ 3. config.py 核心参数详解字典 (调参必看)

这是你未来修改最多的文件。参数分为四大类： 

🛠 A. 系统与硬件配置
参数名,当前值,用途解释 / 修改建议
SHM_NAME,"""shm_ar_video""",共享内存名称，必须与视频采集端（如 C++ 进程）保持一致。
STREAM_PORT,5003,网页图传的端口号。
SEG_CORE,0,"分割模型占用的 NPU 核心 (0, 1, 2)。"
YOLO_CORE,1,检测模型占用的 NPU 核心。
REC_CORE,2,识别模型占用的 NPU 核心。分离核心可避免抢占算力。

📐 B. 图像与逆透视参数
参数名,示例/当前值,用途解释 / 修改建议
TARGET_RES,"(960, 720)",内部处理和网页渲染的统一分辨率。
ROI_TOP_CUT_RATIO,0.3,切除画面顶部 30%。因为天上没有路，切掉可以极大地减少干扰和计算量。
SRC_PTS,"[[0.393, 0.639]...]",(极重要) 逆透视的相机视野 4 个顶点比例。如果车高/摄像头角度变了，必须重新标定这四个点！
DST_PTS,"[[0.385, 0.870]...]",逆透视映射后的物理世界坐标比例，一般不需要改。

🌲 C. 路径搜索与避障参数 (Beam Search)
参数名,当前值,用途解释 / 修改建议
FORK_WIDTH_RATIO,0.35,分叉判定条件 1：赛道总宽度超过画面 35% 认为可能遇到路口。
FORK_GAP_RATIO,0.15,分叉判定条件 2：赛道中间出现了超过 15% 宽度的黑色断层。防止把宽直道误判为分叉。
GAUSSIAN_SIGMA,35.0,YOLO 避障排斥力范围。越大，车子绕开金币的幅度越宽；越小，车子越贴着金币走。
SAFETY_MARGIN,25,避障安全边距（像素）。
SMOOTH_WINDOW,5,路径平滑卷积核大小。如果画出的紫线一直在抖，可以加大到 7 或 9。

🏎 D. 串口控制与 PID 参数
参数名,当前值,用途解释 / 修改建议
SERVO_CENTER,750,舵机中值 (车轮回正时的 PWM)。如果走直道偏一边，微调这个值。
SERVO_MIN/MAX,"590, 910",舵机死区限位，防止机械打死。
MOTOR_MAX_SPEED,2350,直道最高速度 PWM。
MOTOR_STOP,2000,电机停止/刹车 PWM。
KP,0.16,比例系数。控制车头回正的力度。太大容易画龙，太小转不过弯。
KD,160.0,微分(阻尼)系数。用前瞻斜率 l_k 预测未来，抑制过冲画龙。如果出弯时车头摆动，调大它。

🩺 4. 常见问题排查指南 (Troubleshooting)

    画面有红色的路，但没有紫色的引导线？

        原因：找不到起点。可能是画面底部的白线（赛道）太少，触发了 if len(white_ys) > 50: 的拦截。

        解决：检查摄像头是否抬得太高，或者调整 SRC_PTS 把视野拉近。

    遇到分叉路口，车子总是两边摇摆？

        原因：分叉断层检测的阈值太苛刻，导致一帧识别为分叉，下一帧又识别为单路。

        解决：降低 config.py 中的 FORK_GAP_RATIO (如调到 0.10)，让断层更容易被触发。

    终端报错：RuntimeError: load_rknn failed 或找不到核心

        原因：模型路径不对，或 config.py 里变量名没对齐（比如 REC_CORE）。

        解决：检查 config.py 里的 *_MODEL 路径和变量名，确保完全一致。

    网页端 (5003 端口) 非常卡顿？

        原因：网络带宽不够或编码太吃 CPU。

        解决：在 config.py 中把 JPEG_QUALITY 降到 50，或者调低 TARGET_RES 分辨率。不会影响 NPU 实际控制频率。
